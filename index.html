npm-yarn-pnpm

记得刚进木犀的第一件事就是下载各种东西，里面就有node.js（一个大包嘿嘿😘

什么是npm yarn pnpm

既然有了一个资源包，那肯定需要一种手段去管理和利用资源包的内容

  • `npm`就是随`Node.js`一起安装的官方包管理工具，它为开发者搭建了一个庞大的资源库，允许他们在这个平台上搜索、安装和管理项目所必需的各种代码库或模块
  • `yarn`是 Facebook 于2016年发布的一款`JavaScript`包管理器，旨在弥补`npm`在性能和安全性方面的一些缺点,`yarn`引入了 "lockfile" 概念，保证了每次安装的准确性，并使用并行下载来优化性能
  • `pnpm`是另外一个JavaScript包管理器，它的主要卖点是它的高效性，`pnpm` 使用硬盘链接而不是复制来管理项目依赖，这可以极大地节省空间和安装时间

    在执行npm install后，依赖模块被安装到了 node_modules目录下，那node_modules目录下的依赖管理机制又是如何的呢？🤨🤨

    npm Version1(树状结构)

    非常的简单粗暴，npm以递归的形式，严格按照package.json结构以及子依赖包的package.json结构将依赖安装到他们各自的node_modules中,直到有子依赖包不在依赖其他模块

    假如现在我的项目 1.1 依赖两个模块A、B，而模块A又依赖D@1.0.0模块B、C 又依赖D@2.0.0那么，执行完npm i之后, node_modules的结构会和package.json结构一一对应，结构非常明显

    但是，也导致了node_modules会十分庞大：需要安装的模块非常非常多，在不同层级的依赖中，就算是依赖同一个模块，都需要重新安装一次，会有大量的模块重复安装（比如前面的 D@2.0.0 就会安装两次）

    还有可能导致层级嵌套很深：系统对文件路径都会有一个最大长度的限制，嵌套层级过深可能导致奇妙的的问题（比如无法直接删除）

    npm Version3（扁平结构）

    为了解决以上问题，NPM 在版本3做了一次较大更新，将之前的嵌套结构改为扁平结构

    安装模块时，不管其是直接依赖还是子依赖的依赖（间接依赖），优先将其安装在node_modules根目录

    当安装到相同模块时，判断在node_modules根目录已安装的模块版本是否符合新模块的版本范围，如果符合则跳过，不用再次安装；如果不符合就再装一个

    在上一个例子中，那它的结构应该是

    在安装依赖A时，把D@1.0.0优先安装在了node_modules下，接着再安装依赖B时，其依赖的D也会优先考虑安装在根node_modules下，但是根node_modules下已经存在D@1.0.0所以只能将D@2.0.0安装在当前依赖B的node_modules下,最后再安装依赖C@1.0.0也是同样的原则


    虽然在满足Node.js的模块查找规则的同时降低了依赖层级，一定程度上缓解了占用磁盘空间（其实并没有完全解决，比如D@2.0.0还是会被安装两次）和路径过长的问题

    容易引发的问题：结构不确定性和幽灵依赖

    目录结构不确定

    依照 npm V3 的原则，依赖的安装顺序是会改变目录结构的，比如：此时我B先安装，那么目录结构就会变成

    优先安装了B，B又依赖D@2.0.0，所以依赖D@2.0.0 优先安装在根node_modules目录下，当安装依赖A时，其间接依赖D@1.0.0就只能安装在其自身的node_modules目录下


    幽灵依赖：

    一个库使用了没有在package.json中声明的依赖但是可以用，在一开始的例子里，项目 1.1 的依赖可以直接访问到D依赖，虽然它们没有在package.json中声明

    那么就有可能出现：

    不兼容的版本：依赖可能需要的D版本是2.0.0，但是引入的是却是版本1.0.0（这个通过package-lock.json文件可以解决）

    缺少依赖：当我们项目里面没有安装到它所依赖的包时，此时就会报错

    其实幽灵依赖这种隐患在npm树状结构的情况下是能够尽量避免的，因为树状结构严格按照package.json中声明的依赖安装，扁平化这种结构虽然减少依赖重复安装，但它把依赖的依赖提上来之后却使幽灵依赖出现的可能性更大

    npm Version5

    为了解决npm i时依赖结构不确定的问题，在npm版本5新增了package-lock.json文件（一个严格的package.json

  • 固定依赖结构：package-lock.json 的作用是锁定依赖结构，即只要目录下有该文件，那么每次执行npm i后生成的node_modules目录结构是相同的，同时版本也会锁定在指定的范围内

  • 提高安装速度：package-lock.json 中已经缓存了每个包的具体版本和下载链接，不需要再去远程仓库进行查询

    虽然解决了结构不确定性问题，但是同为扁平结构的 npm V5 还是会出现幽灵依赖的问题

    yarn

    yarn是在 2016 年发布的，那时 npm 还处于 V3 时期，还没有package-lock.json文件，就像上面提到的：不稳定性、安装速度慢等缺点经常会受到吐槽，此时，yarn 就诞生了， 后来 npm 也意识到了自己的问题，进行了很多次优化

    yarn 也采用的是 npm v3 的扁平结构来管理依赖，会生成一个yarn.locck文件，不同的是 yarn 在安装包时为并行安装，即：同时下载和安装多个依赖项，而不是按顺序一个接一个地进行，可以显著减少安装时间

    很明显，管理方式使得yarn也会遇到幽灵依赖的情况

    pnpm

    pnpm--performant npm在 2017 年正式发布，定义为快速的，节省磁盘空间的包管理工具，开创了一套新的依赖管理机制，该机制会将包安装在系统的全局 store 中，依赖的每个版本只会在系统中安装一次

    在引用项目 node_modules 的依赖时，会通过硬链接与软链接在全局 store 中找到这个文件

    利用pnpm下载依赖包时，项目的根node_modules里面依赖和package.json里声明的一一对应，简而言之，就是我们在项目的package.json里定义了什么包 modules 文件夹中只会有什么包，不会有次级依赖的包

    回到一开始的例子–项目 1.1 ，那么在根node_modules目录下的依赖只有三个A@1.0.0、B@1.0.0、C@1.0.0，不会有他们的次级依赖，因为在项目 1.1 的package.json中声明的也就只有这三个依赖，没有其他的了

    但是在根node_modules目录下的依赖又不是他们真正的物理位置，而是类似于一个指针，指向.pnpm文件夹下的依赖的真实位置（软链接

    项目的node_modules下会生成一.pnpm文件夹，以平铺的形式储存着所有的包（包括次级依赖），其中，A B C中的依赖D会指向与它们平级的依赖D（软链接），而其本身则通过硬链接指向全局 store 中的真正的依赖包

    当需要引用A时，会指向 modules 文件夹中的那个A，而它会依靠软接指向.pnpm文件夹下的A，进而通过硬链接指向磁盘中真正的依赖包，并且，由于A又依赖于D1，则A下的D1会依靠软链接指向与A平级的D1，再通过硬链接指向磁盘中的D1

    依赖的依赖会在 .pnpm 文件夹中指向同一个平级的依赖，解决了相同依赖重复下载的问题，而由于 modules 文件中并没有次级依赖包，就解决了幽灵依赖的问题，其次，由于引用时路径最终会通过硬链接指向磁盘，使得即时有多个项目也不会重复安装依赖包，节约磁盘空间

    （二）BEM规则

  • 1. 是什么🤨


    是一种命名规范，旨在简化CSS编写并提高代码的可读性和可维护性，能够最有效解决样式之间同名、继承、优先级等所带来的污染性问题，避免最后因为充斥着各种臃肿，而不敢触及修改遗留的各种未知样式

    该方法将HTML元素划分为三个部分:块block元素element和修饰符modifier


    块（block）是BEM的核心部分，它定义了一个HTML元素的基本结构和行为,块可以使用大写字母命名，例如.Header .Main .Footer等,可以随意的嵌套组合、可复用

    元素（element）是块的一个组成部分，它定义了块中特定内容的样式,元素可以使用小写字母命名，例如.header .main .footer等，依赖block存在（出了block就不能用）

    修饰符（modifier）是针对块或元素的特定状态或行为进行配置的方法,修饰符可以使用中划线（_）或破折号（-）进行命名，例如.header--large .main--centered .footer--light等，一个可选的定义block和element外观的行为

    即：利用-和--将样式命名分区😈，不这么写也行，也没人逼你


    BEM优缺点😈

  • 优点：


    结构简单明了易于维护，提高可拓展性

    避免对JS依赖：BEM方法使用静态类名，类名在html中是明确定义的，不需要JavaScript来动态添加或移除类，从而减少了对JavaScript的依赖，降低页面的负载和提高性能

    样式隔离–块独立性：每个块都是独立的，拥有自己的样式，不依赖于其他元素或块，使得块可以在上下文中被使用，更不用担心样式冲突

    样式隔离–元素封装：块内的元素只依赖于它们所属的块，而不是其他块或元素。这确保了元素的样式不会泄露到块外部，保持了样式的封装性

  • 缺点：


    过度使用会觉得代码有点冗余，特别是类名很长的时候（我就爱写长长的类名bushi🥲（只写三级比较好 呃呃🤗

    在项目中拥有大量的块和元素，很有可能导致命名冲突、可拓展性问题

    四、依赖独立性

    这三个字段是JavaScript项目中使用npm管理依赖时package.json文件的组成部分，它们分别代表不同类型的依赖关系：

    （一）dependencies

    是什么：dependencies（依赖）字段列出了项目运行所必需的依赖，这些依赖在项目部署到生产环境时会被包含进来。

    怎么用：当你的项目需要某个库来执行核心功能时，你应该把这个库添加到dependencies中
    例如，如果你的项目是一个web应用，并且使用了React库，那么React就应该被列为dependencies

    示例

    （二）devDependencies

    是什么：devDependencies(开发依赖)字段包含了只在开发过程中需要的依赖，比如测试框架、构建工具、代码分析工具等,这些依赖在生产环境中不会被安装
    怎么用：如果你需要某些工具或库来帮助你开发项目（如Webpack、Jest、ESLint等），但这些工具或库在项目部署后不需要，那么应该将它们添加到devDependencies

    示例

    （三）peerDependencies

    是什么：peerDependencies（协作依赖）字段用于定义你的库需要与其配合使用的其他库的依赖，但不直接指定版本。这通常用于库的作者，当库需要与其他库协同工作，但不打算控制这些库的版本时

    怎么用：当你创建一个库，并希望使用者根据需要安装特定版本的依赖时，你可以在peerDependencies中声明这些依赖。使用者在安装你的库时，需要自己指定这些依赖的版本

    示例

    区别：
    dependencies是项目运行所必需的，会随项目一起部署到生产环境
    devDependencies仅用于开发和测试，不会部署到生产环境
    peerDependencies是一种特殊的依赖声明，用于库与库之间的协同工作，由使用者决定具体版本

    websoket通信连接过程

    服务器和客户端的通信？

    那首先讲讲轮询
    记得上次mini的聊天就是用的轮询🥲🥲

    短轮询
    浏览器每隔一段时间向浏览器发送 http 请求，服务器端在收到请求后，不论是否有数据更新，都直接进行响应，通过让客户端不断的进行请求，使得客户端能够模拟实时地收到服务器端的数据的变化

    这种方式由于需要不断的建立 http 连接，服务器端的压力就会变大

    长轮询
    首先由客户端向服务器发起请求，当服务器收到客户端发来的请求后，服务器端不会直接进行响应，而是先将这个请求挂起，然后判断服务器端数据是否有更新，如果有更新，则进行响应，如果一直没有数据，则到达一定的时间限制才返回

    一般 JavaScript 响应处理函数会在处理完服务器返回的信息后，再次发出请求，重新建立连接

    虽然明显减少了很多不必要的 http 请求次数，但是长时间的连接挂起也会导致资源的浪费

    好，轮到 websoket了😈😈
    WebSocket 是一种基于 TCP 连接上进行全双工通信的协议，相对于 HTTP 这种非持久的协议来说，WebSocket 是一个持久化网络通信的协议；它不仅可以实现客户端请求服务器，同时可以允许服务端主动向客户端推送数据

    websoket通信原理和机制😴😴

    WebSocket 约定了一个通信的规范，通过一个握手的机制，客户端和服务器之间能建立一个类似 tcp 服务器保持长连接，数据发送是双向。

  • 明明 websoket 就是基于 tcp 三次握手应用的，为什么说是类似 tcp 服务器保持长连接？


    TCP（传输控制协议）是一个提供可靠、有序和错误检测能力的传输层通信协议，它通过三次握手建立一个长连接，这个连接可以保持开放状态，允许数据在任意时间双向传输


    WebSocket 协议利用了 TCP 连接的这些特性，但它在应用层上定义了一个新的协议，这个协议通过一个 HTTP 握手过程来升级到 WebSocket 协议。这个过程允许 WebSocket 复用现有的 HTTP 基础设施（如代理、防火墙等），同时拥有其他优势，比如说：全双工通信、实时性、数据传输不需要携带 HTTP 头部信息–数据传输开销减少等等，我们说他 “类似” 是在强调它应用层上的特性

    websoket 建立连接过程😴😴

    websoket 总的来说是三个阶段：打开握手–数据传递–关闭握手
    建立连接时先进行 tcp 三次握手再进行 websoket 两次握手


  • tcp 三次握手🤗

    所有基于 TCP 的连接都需要经过这个过程来确认双方的通信通道已经建立


    最开始的时候客户端和服务器都是处于 CLOSED 关闭状态。主动打开连接的为客户端，被动打开连接的是服务器

    TCP 服务器进程先创建传输控制块 TCB，时刻准备接受客户进程的连接请求，此时服务器就进入了 LISTEN 监听状态


    第一次握手 TCP 客户进程也是先创建传输控制块 TCB，然后向服务器发出连接请求报文，这是报文首部中的同部位 SYN=1，同时选择一个初始序列号 seq=x ，此时，TCP 客户端进程进入了 SYN-SENT 同步已发送状态（证明客户端的发送能力正常）

    第二次握手 TCP 服务器收到请求报文后，如果同意连接，则会向客户端发出确认报文。确认报文中应该 ACK=1，SYN=1，确认号是 ack=x+1，同时也要为自己初始化一个序列号 seq=y，此时，TCP 服务器进程进入了 SYN-RCVD 同步收到状态（证明服务器端的接收能力、发送能力正常）

    第三次握手 TCP 客户端收到确认后，还要向服务器给出确认。确认报文的 ACK=1，ack=y+1，自己的序列号 seq=x+1，此时，TCP 连接建立，客户端进入ESTABLISHED已建立连接状态（触发三次握手，证明客户端的接收能力正常）


    为什么要进行第三次握手？

    客户端向服务器端发送的请求报文由于网络等原因滞留，未能发送到服务器端，此时连接请求报文失效

    客户端在一定时间后没有收到响应，可能会认为请求失败了，因此它会重发相同的连接请求，之后与服务器端建立连接

    当连接释放后，由于网络通畅了，第一次客户端发送的请求报文又突然到达了服务器端，这条请求报文本该失效了，但此时服务器端误认为客户端又发送了一次连接请求，两次握手建立好连接，此时客户端忽略服务器端发来的第二次握手确认请求，并不利用这个连接发送数据，造成不必要的错误和网络资源的浪费


    如果采用三次握手的话，就算那条失效的报文发送到服务器端，服务器端确认并向客户端发送报文，但此时客户端不会发出确认，由于客户端没有确认，由于服务器端没有接收到确认，前面那次不必要的连接请求就不会成功


  • websoket两次握手😈
    使用http请求来完成协议升级过程

    第一次握手：
    客户端通过发送一个HTTP请求到服务器来发起WebSocket握手，这个请求包含特定的HTTP头字段Upgrade: websocket，表明客户端希望将连接从HTTP升级到WebSocket

    第二次握手：
    服务器收到这个HTTP请求后，如果同意升级连接，则会发送一个HTTP响应

    这个响应包含HTTP头字段Upgrade: websocket，表明服务器也同意升级连接

    此外，服务器还会在响应中包含其他一些头字段，如Connection和Sec-WebSocket-Accept，后者是WebSocket协议中的一个安全步骤，用于验证服务器响应的合法性

    websoket断开连接😴😴

  • 首先websoket会用一个控制帧来关闭连接😈
    1. 任何一方（客户端或服务器）都可以通过发送一个包含特定 opcode 的控制帧来发起关闭连接的请求。对于 WebSocket 协议，这个 opcode 是 0x08，代表一个关闭帧（Close Frame）
    2. 一旦一方接收到关闭帧，它必须也发送一个关闭帧作为响应，以确认连接的关闭

  • 一旦 WebSocket 连接关闭，底层的 TCP 连接也会关闭，这就涉及到 tcp 四次挥手😈



    数据传输完毕后，双方都可释放连接。最开始的时候，客户端和服务器都是处于ESTABLISHED状态，然后客户端主动关闭，服务器被动关闭


    第一次挥手 客户端发出连接释放报文，并且停止发送数据。释放数据报文首部，FIN=1，其序列号为seq=u（等于前面已经传送过来的数据的最后一个字节的序号加1），此时，客户端进入FIN-WAIT-1（终止等待1）状态


    第二次挥手 服务器端接收到连接释放报文后，发出确认报文，ACK=1，ack=u+1，并且带上自己的序列号seq=v，此时，服务端就进入了CLOSE-WAIT 关闭等待状态


    第三次挥手 客户端接收到服务器端的确认请求后，客户端就会进入FIN-WAIT-2（终止等待2）状态，等待服务器发送连接释放报文，服务器将最后的数据发送完毕后，就向客户端发送连接释放报文，服务器就进入了LAST-ACK（最后确认）状态，等待客户端的确认


    第四次挥手 客户端收到服务器的连接释放报文后，必须发出确认，ACK=1，ack=w+1，而自己的序列号是seq=u+1，此时，客户端就进入了TIME-WAIT（时间等待）状态，但此时TCP连接还未终止，必须要经过2MSL后（最长报文寿命），当客户端撤销相应的TCB后，客户端才会进入CLOSED关闭状态，服务器端接收到确认报文后，会立即进入CLOSED关闭状态，到这里TCP连接就断开了，四次挥手完成


    为什么客户端要等待2MSL？🤨

    主要原因是为了保证客户端发送那个的第一个ACK报文能到到服务器，因为这个ACK报文可能丢失，并且2MSL是任何报文在网络上存在的最长时间，超过这个时间报文将被丢弃，这样新的连接中不会出现旧连接的请求报文。
